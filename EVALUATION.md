# Evaluation

BitePlate uses the required design patterns in the parts of the system where they fit naturally. The Command pattern is used in the kitchen workflow because kitchen actions are operational commands: start preparing, mark ready, or cancel. Encapsulating each action makes status changes easier to audit, test and undo. A simpler alternative would be direct `if` statements in the view, but that would make the kitchen queue harder to extend when priority changes, undo, or extra kitchen stations are added.

The Singleton order history logger centralises confirmed order logging. This is useful because managers need a consistent source for audit views, date-range reports, table history and frequent-item analysis. The trade-off is that a Singleton can hide dependencies and become too global if it grows. In a larger multi-restaurant deployment, the logger should stay small and delegate database queries to services that always filter by restaurant and branch.

The Strategy pricing implementation keeps discounts away from the bill flow. Standard pricing, happy hour pricing and loyalty pricing can be swapped at runtime without changing the billing code. This is better than hardcoding discount rules in templates or views because restaurants often change promotions. A future production version could store strategy configuration in the database, allowing managers to activate time-based or customer-based pricing without code changes.

Overall, the patterns work together in a coherent restaurant flow: waiters create orders, kitchen commands move orders through the queue, pricing strategies calculate totals, and the history logger records confirmed activity. The current implementation is strong enough for one restaurant or several branches. To scale to 50 restaurants, the most important improvements are tenant-level isolation, background jobs for reports and notifications, stricter permission tests, and stronger operational monitoring for kitchen and POS events.
